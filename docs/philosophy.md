# 哲學：為什麼外部 scaffold 能逼近，又為什麼逼近不全

## 一句話

高階推理模型的可靠度，很大一部分來自它在輸出前的內部「拆解 → 自我糾錯 → 驗證」。這套**流程**可以用外部 hook 重建；這套流程帶來的**品質**只能部分逼近。

## 可以逼近的部分（≈ 80%）：流程紀律

模型常見的失敗不是「不夠聰明」，而是「太快下結論」：

- 自信地給出沒驗證的答案
- 憑記憶描述已經改掉/不存在的東西
- 改一個地方，沒查它的連動
- single-pass 就交付，不自我質疑

這些都是**流程問題**，不是智商問題。外部 scaffold 用 hook 在關鍵時機強制插入一道關卡：
- 要下深度結論前 → 「這是初稿，先跑 Round 2 用 grep/read 驗證每個 claim」
- 要建新東西前 → 「先盤點現有的」
- 要 commit 前 → 「跑 test 了嗎？runtime 驗了嗎？」

強制這些關卡，就補上了大部分「流程紀律」的差距。這部分**任何 model 都吃得到**——Opus、Sonnet、甚至更小的 model 裝上 scaffold，可靠度都會上升。

## 逼近不了的部分（≈ 20%）：原生能力

scaffold 能逼模型「去驗證」，但逼不出：
- **洞察深度**：一次就看穿「這 21 個 bug 其實是 5 群同根症狀」的能力
- **驗證品質**：同樣跑 Round 2，高階 model 找到的反例更準、更刁
- **效率**：高階 model 的內生思考有底層硬體優化（KV cache、並行），外部 prompt 硬逼思考是「逐字生成」，慢且貴

這 20% 是 model 本身的事，scaffold 補不了。

## 代價：時間與 token

強制思考 = 多生成 token = 更慢、更貴，常見 2-3 倍。所以：

- **只在難任務上開**：複雜除錯、多檔案重構、深度研究、架構決策。
- **簡單問答關掉**：問「今天星期幾」不需要跑 Round 1-3。
- **動態路由**：理想做法是有個輕量判斷層，難任務才啟動完整 scaffold（見 `architecture.md` 環節 5）。

## 結論

把 scaffold 想成給車加渦輪：馬力（可靠度）上升，代價是耗油（token）和引擎發熱（延遲）。**只在陡坡踩油門**，是最經濟的玩法。它不會把你的車換成跑車（model 還是同一個），但能讓你在難路段開得更穩。
